# AI 小说试读决策与续读恢复助手

本文件是项目的详细设计说明，当前实现版说明见同目录 `README.md`。

## 1. 产品定位

项目从“AI 个性化小说推荐系统”收敛为：

> 帮重度小说用户少踩烂书，快速接回断更书。

它不再把推荐流作为唯一主角，而是围绕两个更明确的用户问题展开：

- 新书到底值不值得看？
- 老书很久没看，怎么快速接上剧情？

## 2. 用户痛点

### 简介很好看，但正文不一定好看

网文简介和标题往往承诺强设定、强爽点、强悬念，但正文前三章可能出现节奏拖、主角不讨喜、设定不兑现、套路陈旧等问题。用户需要一个低成本的“前三章止损”工具。

### 长篇追更容易断层

免费阅读用户经常同时追很多书，长篇读到几百章后隔一段时间再回来，很容易忘记人物关系、主角状态、当前冲突和主线推进。普通摘要不能很好地回答“我现在该从哪里接着看”。

### 推荐列表缺少解释

单纯推荐“你可能喜欢”不够。用户更关心：这本书为什么适合我？可能哪里不适合？我应该先读几章验证什么？

## 3. 核心功能设计

### A. 试读判定

用户选择本地 TXT，默认分析前三章。

输出：

- 前三章讲了什么
- 主角人物形象
- 剧情推进到哪里
- 简介/标题承诺是否兑现
- 是否值得继续读

产品价值：把“读十几章才发现不好看”的成本压缩到一次 AI 判定。

### B. 续读恢复

用户输入“我读到第 N 章”。

输出：

- 前情讲到哪里
- 主角当前状态
- 当前剧情阶段
- 关键冲突和未解决问题
- 接下来怎么接着读

产品价值：让长篇追更用户不用重新翻几十章找记忆点。

### C. 个性化推荐

推荐能力作为辅助：

- 书架内候选：解决“书架里哪些值得继续看”。
- 书架外候选：解决“还有哪些可能值得试读”。

推荐结果需要提供来源、命中标签、风险点和试读计划，而不是只展示分数。

## 4. 页面信息架构

当前界面保留三个主入口：

```text
试读判定
续读恢复
个性化推荐
```

原“章节记忆库”不再作为用户主入口，而是底层能力。面试时可以说明：系统内部会把章节拆成事件、人物、伏笔和记忆块，但用户只看到“是否值得读”和“怎么接着读”。

## 5. LLM 分块方案

### 短文本试读

前三章文本量可控，直接送入 LLM，并要求输出结构化 JSON：

- plot_summary
- protagonist_profile
- hook_delivery
- risks
- verdict

### 长篇续读

1-N 章不能一次性塞给模型，需要多级压缩：

1. 章节解析
2. 分块摘要
3. 阶段合并
4. 最终续读恢复

最终用户只看到简洁的续读结果，内部的“60 个章节记忆块、12 个剧情阶段”等信息不直接展示。

## 6. 数据边界

- 不抓取盗版章节。
- 不绕过平台限制读取未授权内容。
- 正文来自用户本地有权阅读的 TXT。
- 番茄书架数据来自用户主动导入或本地脱敏样本。
- 书架外候选来自公开候选池和公开页面，不声称拥有番茄内部全站推荐能力。

## 7. 面试亮点

可以这样表述：

> 这个项目不是简单做推荐列表，而是把免费阅读产品里的弃书成本、追更断层和推荐不可信三个问题，转成了可交互的客户端 AI 能力。试读判定帮助用户少踩烂书，续读恢复帮助用户快速回到长篇剧情，推荐模块则为这两个决策提供候选和解释。

## 8. 当前实现文件

```text
app/fanqie_recommender.py     书架内候选重排
app/public_recall.py          书架外公开候选池
static/index.html             三入口界面
static/app.js                 前端状态和接口调用
static/styles.css             页面样式
README.md                     运行说明和项目介绍
```
