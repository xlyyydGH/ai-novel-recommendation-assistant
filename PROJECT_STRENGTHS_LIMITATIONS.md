# 项目优势与缺陷分析

本文档用于面试前复盘，帮助说明该项目相比普通推荐系统 Demo 的亮点、当前边界和后续优化方向。

## 一、项目优势

### 1. 问题定义贴近真实阅读场景

项目不是泛泛地做“小说推荐”，而是从网文用户的真实痛点出发：

- 简介看起来很好，但读几章发现正文不对味。
- 长篇小说追更时间长，容易忘剧情、忘人物关系、忘伏笔。
- 书架里书很多，但不知道哪些值得继续看。
- 用户不只是缺书，更缺少能降低试错成本的阅读助手。

这类问题和番茄小说的阅读、推荐、留存体验强相关，比普通 CRUD 或静态推荐页更贴近业务。

### 2. 接入真实书架画像，而不是只用模拟数据

项目已经引入脱敏后的番茄书架数据，用真实书架里的书名、进度、状态和题材标签构建用户画像。

其中一个重要设计是：**低网页进度不等于不喜欢**。网页端进度可能未同步，也可能只是刚加入书架，所以系统不会直接把低进度当作负反馈，而是标记为“进度待验证”。这体现了对真实产品数据噪声的理解。

### 3. 推荐链路可解释

每本候选书会展示：

- 命中的高热爱标签
- 召回来源
- 画像匹配分
- 标题兴趣分
- 试读风险
- 正负反馈影响
- 为什么适合或可能不适合

这让推荐结果不是黑盒，而是能帮助用户判断“为什么先看这本”。对客户端岗位来说，这类解释性展示也有利于转化成产品交互。

### 4. 区分书架内续读和书架外发现

当前页面将推荐分成两类：

- 书架内待验证推荐：来自真实书架，解决“我已有书架里哪些值得继续看”。
- 书架外公开候选池：来自公开页面和公开候选，解决“我还能发现什么新书”。

这比把所有候选混在一个推荐流里更清楚，也更符合真实阅读产品的使用习惯。

### 5. AI 阅读助手能力与小说业务强绑定

项目中的 AI 能力不是泛用聊天，而是围绕小说阅读体验展开：

- 无剧透选书报告
- 正文导入后的剧情记忆卡与正文级试读判断
- 本地 TXT 指定章节试读分析：从真实 TXT 样本读取正文，支持起始章和分析章数，生成剧情回忆、人物关系、伏笔和无剧透试读判断
- MiniMax M2.7 深度理解：规则层切章，LLM 层做人设关系、机缘道具、伏笔追踪和无剧透报告，体现模型路由与成本控制意识
- 续读前情回顾
- 章节摘要
- 人物关系梳理
- 伏笔提醒
- 简介兑现度判断

这些功能能直接解决长篇追更和选书试错的问题，和番茄小说业务契合度较高。

### 6. 有显式反馈闭环

项目支持“想看类似”“不感兴趣”“简介不符”“节奏太慢”“主角降智”等反馈，并把反馈写入本地画像，影响后续排序。

这说明项目不是一次性展示推荐结果，而是考虑到了用户反馈、画像更新和排序校准。

### 7. 面试表达空间大

该项目可以覆盖多个面试话题：

- 用户洞察
- 客户端交互设计
- 推荐系统基础链路
- 数据噪声处理
- 可解释推荐
- AI 阅读助手落地
- 反馈闭环
- 无内部数据权限时的工程取舍

这比单纯展示一个页面或调用一个 LLM API 更有层次。

## 二、当前缺陷

### 1. 书架外候选池仍然较小

由于无法访问番茄内部推荐后台，项目不能获得真实全站候选、用户行为日志、曝光点击数据和工业级召回结果。

当前书架外候选主要来自公开页面、公开关键词页和本地种子候选。因此不能说“实现了番茄全站推荐”，更准确的说法是：

> 在无内部数据权限条件下，实现了公开候选召回与本地个性化重排。

### 2. 推荐算法以规则和启发式为主

目前排序主要依赖标签匹配、关键词命中、来源权重、试读风险和显式反馈，还不是工业级推荐模型。

缺少的能力包括：

- 向量召回
- 协同过滤
- 学习排序模型
- 多目标排序
- 在线 A/B 实验
- 大规模曝光、点击、完读、弃读日志

因此它更适合定位为 MVP 和业务原型，而不是完整工业推荐系统。

### 3. AI 能力部分仍受数据限制

当前“续读前情回顾”主要基于书架进度、题材标签和高热爱画像生成。由于没有批量抓取真实章节正文，系统不会编造具体剧情。

这是一种安全边界，但也意味着：

- 前情回顾还不能做到真实剧情级回忆。
- 人物关系梳理需要更多正文输入。
- 未导入章节正文时，无剧透选书报告仍依赖公开简介和元数据，深度有限。
- 导入用户有权阅读的章节正文后，系统可以生成剧情记忆卡和正文级试读判断，但不会自动抓取未授权章节，也不会保存章节原文。
- TXT 分析能力目前面向本地合法样本库，适合展示“基于正文阅读后的分析链路”；如果要接入线上书源，需要明确版权授权、用户授权和平台接口边界。
- MiniMax 深度分析依赖外部 API Key 和网络，面试演示时可以先用缓存结果或规则层报告兜底，避免现场网络波动影响展示。

### 4. 缺少长期反馈验证

虽然有反馈按钮，但目前还没有真实验证：

- 用户点击推荐后是否继续阅读
- 是否读完前三章
- 是否加入书架
- 是否后续弃读
- 反馈后推荐命中率是否提升

因此推荐效果仍停留在可解释 Demo 阶段，缺少长期指标闭环。

### 5. 前端仍是面试 Demo 形态

当前页面适合演示项目逻辑，但还不是成熟移动端阅读产品。

缺少的体验包括：

- 移动端沉浸式阅读页
- 真实封面和书籍详情页
- 推荐流滚动加载
- 阅读器内划线、摘要、人物卡片
- App 端续读入口
- 更自然的反馈交互

如果面试客户端岗位，可以强调当前重点是业务闭环和交互原型，后续可继续做移动端体验升级。

## 三、后续优先改进

### 1. 接入真实章节文本

优先给少量书导入前几章或公开样本文本，让前情回顾、人物关系和简介兑现度分析真正基于正文，而不是只基于元数据。

### 2. 增加推荐效果评估面板

展示反馈前后：

- 哪些标签权重上升或下降
- 哪些候选被过滤
- Top 候选分数如何变化
- 试读风险如何变化

这会让项目更像推荐系统，而不只是展示页。

### 3. 增加语义召回

可以用 embedding 对书名、简介、标签、用户高热爱样本做向量化，补充当前关键词匹配的不足。

### 4. 强化移动端体验

如果面向客户端岗位，可以继续做：

- 移动端阅读页
- 前情回顾浮层
- 人物关系卡片
- 章节摘要抽屉
- 推荐卡片滑动反馈

## 四、面试推荐表述

可以这样概括项目：

> 这个项目不是复刻番茄内部推荐后台，而是在无内部数据权限的条件下，从真实小说用户痛点出发，实现了一个个性化推荐与 AI 阅读助手 MVP。项目用脱敏书架数据构建高热爱画像，区分书架内续读和书架外发现，并通过可解释推荐、无剧透选书报告、正文导入后的剧情记忆卡、续读前情回顾和显式反馈闭环，降低用户选书试错成本和长篇追更记忆负担。

如果面试官问“和番茄真实推荐系统差距在哪”，可以回答：

> 差距主要在候选规模、真实行为日志和模型复杂度。真实推荐系统会有全站候选、曝光点击完读等大规模日志、多路召回、排序模型和 A/B 实验；我的项目目前是无后台权限下的 MVP，所以重点放在公开候选召回、真实书架画像、可解释推荐和 AI 阅读体验增强上。它更像一个产品原型和推荐链路证明，而不是工业级推荐系统。
